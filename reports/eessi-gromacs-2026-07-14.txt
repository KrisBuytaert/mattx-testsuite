Notes / known intermittent issue
---------------------------------
An earlier run of this same test (same cluster, same commit) hit a
different failure: the migrated gmx process crashed with
"traps: gmx[N] trap invalid opcode" immediately after being resumed on
the target node (dmesg: "IT'S ALIVE! Sending SIGCONT" followed
immediately by the invalid-opcode trap), which then triggered a real
process-exit/funeral on the source node. This did NOT reproduce on the
run documented above. This looks like an upstream MattX kernel-module bug
in FPU/AVX register-state restoration during migration (the EESSI GROMACS
build used is AVX2-optimized "haswell") — distinct from the already-filed
mt-8i8 (Guest Watcher race / premature-death false detection). Filed as
a new bug for upstream MattX to investigate; see bd for bug ID. This is
intermittent, not consistently reproducing, so it did not block this run,
but reruns of this test may occasionally hit it.

Fixes made to the testsuite scripts to get this test running (see git log,
commit "fix: unblock Debian cluster provisioning (mt-985.3)" and the
GMX_PID capture fix in scripts/test-eessi-gromacs.sh):
- scripts/setup-node.sh: apt dpkg-lock wait loop checked the wrong lock.
- scripts/sources.conf: MATTX_SRC pointed at a nonexistent path.
- scripts/start-mattx.sh: referenced a nonexistent mattx-discd.service unit
  (the installed unit is named mattx.service).
- scripts/test-eessi-gromacs.sh: GMX_PID capture picked up EESSI/Lmod init
  banner text from stdout instead of isolating the trailing `echo $!`,
  corrupting every later kill/ps use of the PID.
